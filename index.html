Overview

여러 판매자가 한 플랫폼에 입점하는 멀티 셀러 오픈마켓입니다. 판매자가 등록한 상품을 관리자가 승인해야 노출되고, 한 주문에 여러 판매자 상품이 섞이므로 배송과 정산을 판매자별로 분리해야 합니다. 여기에 PortOne V2 결제·정산·관리자 KPI 대시보드까지 이커머스 전체 흐름을 1인 풀스택(프론트·백·인프라)으로 구현했습니다. 본 포트폴리오는 프론트엔드 직무 기준으로 서술하며, 백엔드·인프라는 보조로 둡니다.

02 AI Object Detection Viewer
AI 객체 탐지(COCO) 결과를 2D·3D에서 동시에 탐색하고, 4개 뷰가 단일 선택 상태를 공유하는 멀티뷰 동기화 시각화 뷰어.

Overview

객체 탐지 결과(COCO)를 보통 2D 박스로만 보는 것과 달리, 같은 결과를 2D 이미지와 3D 공간에서 동시에 탐색합니다. 한 객체를 어느 뷰에서 선택하든 4개 뷰(2D·3D·리스트·분석)가 동시에 같은 객체를 가리키도록 설계했습니다. 데이터 소스가 정적 JSON 1개인 클라이언트 SPA입니다 — R3F Canvas가 SSR 불가라 전 페이지 클라이언트 렌더.

시스템 구조

정적 COCO JSON → lib/ 순수 함수 계층(parser·estimator·selector)이 외부 입력을 unknown으로 받아 isCocoDataset 타입 술어로 검증 후 내부 Frame[]로 변환 → Zustand store(단일 선택 상태)를 4뷰가 각자 필요한 slice만 구독. lib/와 components/를 분리해 React 밖에서 JSDOM 없이 테스트하고 변경 영향을 격리했습니다.

① Firebase 인증 구독이 매 렌더마다 재등록 렌더/부수효과 경계
PProblem전역 AuthGuard에서 페이지를 오갈 때마다 로그인 정보가 미세하게 깜빡이고, 라우트를 전환할수록 불필요한 리렌더가 늘었다. React DevTools로 보면 인증 리스너가 컴포넌트가 리렌더될 때마다 하나씩 늘어나 정리되지 않았다.
AAnalyzeonAuthStateChanged는 "한 번 등록하고 cleanup해야 하는 구독(subscribe) API"인데, useEffect 없이 컴포넌트 본문에서 직접 호출되고 있었다. 그 결과 렌더마다 새 리스너가 등록되고 이전 리스너는 해제되지 않아 누적됐고, 누적된 리스너들이 인증 상태 변화마다 각각 setUser를 호출해 중복 리렌더와 깜빡임을 일으켰다. 반환된 unsubscribe를 아무도 쓰지 않는다는 점이 결정적 단서였다.
AAction구독을 useEffect 내부로 옮기고 반환된 unsubscribe를 cleanup으로 등록해 마운트 시 1회 등록 · 언마운트 시 해제되도록 했다. 재발 방지로 react-hooks/exhaustive-deps 규칙을 warning에서 error로 승격해, deps 배열 누락 시 빌드가 막히도록 했다.

배운 점 · 외부 구독은 렌더가 아니라 **라이프사이클(useEffect)**에서 관리해야 한다. cleanup 미등록은 리스너 누적·다중 setState를 부른다. 구독 API가 반환하는 unsubscribe를 쓰지 않고 있다면 거의 항상 잘못된 위치에서 호출되고 있다는 신호다.

AAnalyze예약 정보를 담은 Recoil bookingAtom은 메모리에만 존재 → 새로고침 시 null로 초기화된다. 코드가 null 검사 없이 곧바로 구조분해(bookingData as BookingData)해 Cannot destructure property 'from' of null 에러가 났다. history.back() 가드 effect가 있었지만, 그 effect가 실행되기 전에 구조분해가 먼저 터지는 실행 순서 문제라 무력했다.
AAction1차로 모든 Hook 호출 이후에 null 가드(<Navigate>)를 두어 크래시를 막고 홈으로 안전하게 리다이렉트했다 — 가드를 Hook 사이가 아닌 끝에 배치해 Rules of Hooks("Rendered fewer hooks" 에러) 위반을 피했다. 2차로 새로고침에도 플로우가 유지되도록 bookingAtom을 effects_UNSTABLE로 sessionStorage와 동기화했다(Date 필드는 복원 시 reviver로 되살림).
RResult
새로고침·URL 직접 진입에도 크래시 없이 동작하고, 진행 중이던 예약 정보가 복원된다.
배운 점 · SPA의 전역 상태는 휘발된다. 영속화(sessionStorage)/URL 복원 중 데이터 성격에 맞는 방식을 골라야 한다. 결제 정보는 일시적·민감해 탭을 닫으면 사라지는 sessionStorage를 택했다. 또한 조기 return(가드)은 항상 모든 Hook 호출 이후에 둬야 Hook 순서가 깨지지 않는다는 점을 체득했다.
